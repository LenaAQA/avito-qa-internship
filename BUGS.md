# BUGS

## BUG-001 — Поле statistics обязательно при создании объявления
* Документация (Postman-коллекция) также содержит некорректный контракт — поле `statistics` обязательно
**Описание:**  
При создании объявления (`POST /api/1/item`) поле `statistics` является обязательным.  
Без него API возвращает ошибку 400 "поле likes обязательно". 


**Шаги воспроизведения:**
1. Отправить POST /api/1/item  
2. Тело запроса:

    {
      "sellerID": 5566,
      "name": "test",
      "price": 6
    }

**Фактический результат:**  
400 Bad Request  

    {
      "result": {
        "message": "поле likes обязательно"
      },
      "status": "400"
    }

**Ожидаемый результат:**  
Поле `statistics` не должно быть обязательным при создании объявления,  
так как статистика (likes, viewCount, contacts) — это производные данные,  
которые должны формироваться системой, а не передаваться пользователем.

**Серьёзность:** P1 (высокий)  
**Приоритет:** P1  

**Окружение:**  
https://qa-internship.avito.com

## BUG-002 — GET /item/{id} возвращает список вместо объекта
* Документация (Postman-коллекция) также содержит некорректный контракт — указан массив вместо объекта

**Описание:**  
Метод получения объявления по id (`GET /api/1/item/{id}`) возвращает массив,  
хотя по логике должен возвращать один объект.

**Шаги воспроизведения:**
1. Создать объявление  
2. Выполнить GET /api/1/item/{id}

**Фактический результат:**  

    [
      {
        "id": "...",
        "sellerId": 5566,
        "name": "...",
        "price": 6,
        "statistics": {
          "contacts": 6,
          "likes": 6,
          "viewCount": 6
        }
      }
    ]

**Ожидаемый результат:**  

    {
      "id": "...",
      "sellerId": 5566,
      "name": "...",
      "price": 6,
      "statistics": {
        "contacts": 6,
        "likes": 6,
        "viewCount": 6
      }
    }

**Серьёзность:** P1 (высокий)  
**Приоритет:** P1  

**Комментарий:**  
Нарушение REST-контракта: получение по ID должно возвращать единичный ресурс, а не коллекцию.

## BUG-003 — GET /statistic/{id} возвращает список вместо объекта
* Документация (Postman-коллекция) также содержит некорректный контракт — указан массив вместо объекта

**Описание:**  
Метод получения статистики (`GET /api/1/statistic/{id}`) возвращает массив,  
хотя статистика относится к одному объявлению.

**Шаги воспроизведения:**
1. Создать объявление  
2. Выполнить GET /api/1/statistic/{id}

**Фактический результат:**  

    [
      {
        "contacts": 6,
        "likes": 6,
        "viewCount": 6
      }
    ]

**Ожидаемый результат:**  

    {
      "contacts": 6,
      "likes": 6,
      "viewCount": 6
    }

**Серьёзность:** P2 (средний)  
**Приоритет:** P2  

**Комментарий:**  
Нарушение контракта: статистика — это 1:1 сущность, а не список.

## BUG-004 — Ответ POST /item не соответствует документации

**Описание:**  
Метод создания объявления (`POST /api/1/item`) возвращает тело ответа, не соответствующее документации.

**Шаги воспроизведения:**
1. Отправить POST /api/1/item с валидным телом

**Фактический результат:**  

    {
      "status": "Сохранили объявление - <id>"
    }

**Ожидаемый результат:**  

    {
      "id": "<string>",
      "sellerId": "<integer>",
      "name": "<string>",
      "price": "<integer>",
      "statistics": {
        "likes": "<integer>",
        "viewCount": "<integer>",
        "contacts": "<integer>"
      },
      "createdAt": "<string>"
    }

**Серьёзность:** P0 (критический)  
**Приоритет:** P0  

**Комментарий:**  
- Невозможно использовать ответ для последующих шагов (E2E)  
- Нарушен контракт API  
- Ломает автоматизацию (нельзя валидировать response)
